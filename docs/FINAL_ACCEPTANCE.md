# 阶段性终点验收

状态：草案，随 v2.0-v2.2 实现持续更新
目标版本：v2.2.5

## 目的

本文档定义 HaoShortLink 当前阶段的完成条件。Kubernetes 或 Transactional Outbox 的代码合入本身不等于
项目完成；只有产品、运行、可靠性、测试、文档和演示证据同时闭环，才进入 v2.2.5 功能冻结。

阶段终点应满足六个关键词：

- 可用：用户能登录、创建和管理自己的短链接，公共访问者能稳定跳转。
- 可见：管理页面、统计、生命周期审计、指标和日志能展示系统行为。
- 可运行：Compose 和 Kubernetes 都有可重复的启动、检查和清理入口。
- 可恢复：缓存、消息系统、进程或 Pod 故障后，核心语义和积压恢复符合设计。
- 可验证：功能、安全、迁移、性能、长稳和故障测试有可复现结果。
- 可解释：每个中间件都有明确职责、故障边界、取舍和不使用它的后果。

## 完成定义

### v2.0 产品闭环

- 用户注册、登录、退出、会话过期 / 撤销和当前用户查询可用。
- 链接具备 owner，对列表、详情、更新、禁用 / 恢复、过期设置和统计查询执行对象级授权。
- 公共跳转保持匿名可用，普通用户不能读取或修改他人的链接。
- 自定义短码处理格式、保留路径、唯一约束、并发冲突和权限边界。
- 最小管理页面完成登录、创建、列表、生命周期管理、统计和错误反馈闭环。
- OpenAPI 与实际 API 一致；空库初始化和从 v1.9 数据升级均有自动化验证。
- 密码、会话 token、日志、指标和错误响应满足已记录的安全边界。

### v2.1 Kubernetes 运行闭环

- 固定一种本地 Kubernetes 环境，并记录版本、资源和依赖拓扑。
- 服务、访问事件 consumer 和管理入口具备可复现镜像、Deployment、Service、配置、Secret 与探针。
- MySQL 持久化模式下服务多副本语义正确；consumer group 扩展不重复累计统计。
- 内部 API、指标和依赖端口没有通过公共入口意外暴露。
- 滚动升级、回滚、Pod 删除重建和关键依赖故障均有可重复验收步骤。
- 启动、演示、健康检查和清理入口已在干净环境运行成功。

### v2.2 可靠事件闭环

- 链接生命周期业务变更与 outbox 记录在同一 MySQL 事务中提交或回滚。
- 独立 relay 能并发安全领取记录、发布、退避重试、优雅停止、恢复积压并执行受控清理。
- 生命周期 consumer 使用 `event_id` 幂等生成审计投影，管理 API / 页面可以查询时间线。
- 访问事件继续保持异步 fail-open，不为跳转增加同步 outbox 写入。
- 发布成功但状态回写失败、Kafka 中断、relay / consumer 崩溃和重复投递均有明确结果与测试证据。
- outbox 积压量、最老待发布年龄、发布 / 重试 / 失败和消费结果具备低基数观测。
- 文档明确系统是至少一次交付与幂等消费，不声明端到端 exactly-once。

## 必须通过的演示路径

一次完整演示至少覆盖：

1. 从干净环境启动依赖、应用、consumer、relay、监控和管理入口。
2. 注册 / 登录，创建随机或自定义短码，匿名访问短码并看到访问统计异步更新。
3. 禁用、恢复或修改过期时间，并在生命周期审计时间线看到对应事件。
4. 在 Prometheus / Grafana 或指标入口观察请求、缓存、限流、Kafka、consumer 和 outbox 状态。
5. 删除一个服务 Pod，确认流量通过其他副本继续处理并完成重建。
6. 中断 Redis，确认查询缓存和创建限流按既定 fail-open / 回源语义运行并产生观测。
7. 中断 Kafka，确认跳转不被阻塞、生命周期 outbox 形成积压；恢复 Kafka 后积压被处理且审计不重复。
8. 执行清理命令，不遗留无法解释的进程、容器或测试数据。

## 最终测试矩阵

| 类别 | 最低覆盖 | 通过标准 |
| --- | --- | --- |
| 单元与组件 | 框架、业务规则、认证、授权、自定义短码、事件 codec、relay 状态机 | 确定性通过，无外部依赖的测试可快速执行 |
| API 与浏览器 E2E | 注册登录、管理闭环、匿名跳转、统计、审计和错误反馈 | 从用户入口完成闭环，不依赖手工改库 |
| 安全 | 密码 / token、对象越权、输入边界、保留路径、内部端口、敏感日志 | 无已知高风险越权或凭据泄露 |
| 数据迁移 | 空库、v1.9 升级、回填、重复执行、失败回滚 | 数据可用且 schema 版本可追踪 |
| 依赖集成 | MySQL、Redis、Kafka、Nginx / Ingress、Prometheus / Grafana | 既定成功、降级和 readiness 语义全部一致 |
| Kubernetes | 多副本、探针、配置、Secret、发布、回滚、Pod 与依赖故障 | 服务恢复且无未解释的数据重复或丢失 |
| Outbox | 原子性、崩溃窗口、重试、重复、积压、恢复、清理 | 业务与事件不出现双写分叉，审计投影保持幂等 |
| 性能与长稳 | 基线、阶梯升压、30-60 分钟稳定性、带故障负载 | 结果可复现，无崩溃、失控积压或持续错误增长 |
| 干净环境 | 源码构建、迁移、Compose、Kubernetes、演示和清理 | 只依赖文档即可复现，不使用个人机器遗留状态 |

## 性能与稳定性记录要求

- 固定 git commit、工具版本、VM / Kubernetes 资源、镜像、配置、数据规模和预热方式。
- 每个核心场景至少执行三轮，记录 QPS、平均延迟、P50、P95、P99、错误率、CPU、内存和连接池状态。
- 同时记录 Kafka consumer lag、outbox backlog、最老待发布年龄以及故障恢复耗时。
- 至少覆盖 Redis 命中跳转、MySQL 回源、短码不存在、认证后创建、管理查询和混合负载。
- 对比单副本与双副本，并执行一次 Redis 或 Kafka 故障下的持续负载。
- 本地结果只作为固定环境中的相对基线，不包装成生产承载上限。

## 发布检查

- Linux VM Release 构建、CTest、脚本语法、API / 集成 / 故障测试和浏览器 E2E 全部通过。
- Compose 与 Kubernetes 两条演示链路均通过，且完成独立干净目录验证。
- 代表性性能、长稳和带故障负载结果已记录；性能测试不设置为普通 PR 的硬门禁。
- README、架构、API、数据模型、运行、部署、测试、压测、故障处理和路线图与代码一致。
- 未实现能力没有写成已完成，已知限制与恢复边界有明确说明。
- v2.2.5 发布后冻结本阶段功能范围，只接受缺陷、安全、文档和必要维护变更。

## 不属于终验的能力

Go、RabbitMQ、Schema Registry、Elasticsearch / ClickHouse、Terraform、公有云、生产级有状态集群、
Service Mesh、自定义域名、OIDC、邮件和计费均不是 v2.2.5 的完成条件。是否引入必须由新的真实需求决定。
