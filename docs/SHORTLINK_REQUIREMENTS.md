# 短链接需求

状态：草案
当前实现：已实现健康检查、创建短链接、短码跳转、MySQL 持久化和可选 Redis 查询缓存

## V1 目标

ShortLink V1 的目标是实现短链接服务的最小闭环：

- 服务进程可以作为真实 HTTP 服务启动。
- 服务健康检查。
- 创建短链接。
- 根据短码跳转到原始 URL。

V1 的验收标准是：

```text
服务可启动 -> 可创建短链 -> 可访问短链并返回 302 跳转
```

## 核心概念

短链接服务维护一组映射关系：

```text
短码 -> 原始 URL
```

用户访问短链接时，服务根据短码找到原始 URL，并返回 HTTP 重定向响应。

V1 默认使用进程内存保存映射关系。v1.1 已补充 MySQL 持久化和可选 Redis 查询缓存。

## V1 能力

### 健康检查

用于确认服务进程和 HTTP 路由是否可访问。

当前状态：已实现。

接口：

```text
GET /api/health
```

成功响应：

```json
{
  "status": "ok"
}
```

### 创建短链接

客户端提交原始 URL，服务生成短码并返回短链接信息。

当前状态：已实现。

接口：

```text
POST /api/short-links
```

V1 最小要求：

- 请求体使用 JSON。
- 请求字段包含 `url`。
- `url` 至少需要是 `http://` 或 `https://` 开头的 URL。
- 创建成功后返回短码、短链路径和原始 URL。
- 参数缺失、JSON 无效或 URL 不符合最小规则时返回 400 JSON 错误。

### 短码跳转

用户访问短码路径，服务查找原始 URL，并返回重定向响应。

当前状态：已实现。

接口：

```text
GET /s/{code}
```

默认跳转状态码：

```text
302 Found
```

V1 最小要求：

- 短码存在时返回 `302 Found`。
- 响应头 `Location` 指向原始 URL。
- 短码不存在时返回 404 JSON 错误。

## V1 暂不包含

- 用户系统。
- 链接列表查询。
- 链接删除或更新。
- 访问统计。
- 过期策略。
- 限流和风控。
- MySQL 持久化。
- Redis 缓存。
- Docker Compose。
- Nginx。
- 压测。
- 复杂短码策略。
- 完整自动化测试体系。

## v1.1 目标

v1.1 的目标是在 V1 最小闭环基础上补充持久化和查询缓存。

验收标准：

```text
创建短链 -> 服务重启 -> 已创建短链仍可跳转
```

已实现能力：

- 使用 MySQL 持久化短码到原始 URL 的映射。
- 使用 MySQL 自增 id 生成 Base62 短码，避免服务重启后进程内计数器重置导致撞码。
- 使用 Redis 缓存短码跳转查询结果。
- Redis 只优化查询，不改变业务语义；短链是否存在和跳转到哪里以 MySQL 为准。
- Redis 未命中时继续查询 MySQL。
- Redis 不可用时继续回源 MySQL，不应把已有短链误判为不存在。

v1.1 不改变 V1 对外 API：

```text
GET  /api/health
POST /api/short-links
GET  /s/{code}
```

## v1.1 暂不包含

- 访问统计。
- 过期策略。
- 限流和风控。
- Docker Compose。
- Nginx。
- 压测。
- 消息队列。
- 监控和告警。
