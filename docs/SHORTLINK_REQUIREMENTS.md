# 短链接需求

状态：草案
当前实现：已实现 V1 内存版健康检查、创建短链接和短码跳转

## V1 目标

ShortLink V1 的目标是实现短链接服务的最小闭环：

- 服务进程可以作为真实 HTTP 服务启动。
- 服务健康检查。
- 创建短链接。
- 根据短码跳转到原始 URL。

V1 的验收标准是：

```text
服务可启动 -> 可创建短链 -> 可访问短链并返回 302 跳转
```

## 核心概念

短链接服务维护一组映射关系：

```text
短码 -> 原始 URL
```

用户访问短链接时，服务根据短码找到原始 URL，并返回 HTTP 重定向响应。

V1 使用进程内存保存映射关系。服务重启后，已创建短链接会丢失。持久化和缓存属于后续版本。

## V1 能力

### 健康检查

用于确认服务进程和 HTTP 路由是否可访问。

当前状态：已实现。

接口：

```text
GET /api/health
```

成功响应：

```json
{
  "status": "ok"
}
```

### 创建短链接

客户端提交原始 URL，服务生成短码并返回短链接信息。

当前状态：已实现。

接口：

```text
POST /api/short-links
```

V1 最小要求：

- 请求体使用 JSON。
- 请求字段包含 `url`。
- `url` 至少需要是 `http://` 或 `https://` 开头的 URL。
- 创建成功后返回短码、短链路径和原始 URL。
- 参数缺失、JSON 无效或 URL 不符合最小规则时返回 400 JSON 错误。

### 短码跳转

用户访问短码路径，服务查找原始 URL，并返回重定向响应。

当前状态：已实现。

接口：

```text
GET /s/{code}
```

默认跳转状态码：

```text
302 Found
```

V1 最小要求：

- 短码存在时返回 `302 Found`。
- 响应头 `Location` 指向原始 URL。
- 短码不存在时返回 404 JSON 错误。

## 暂不包含

- 用户系统。
- 链接列表查询。
- 链接删除或更新。
- 访问统计。
- 过期策略。
- 限流和风控。
- MySQL 持久化。
- Redis 缓存。
- Docker Compose。
- Nginx。
- 压测。
- 复杂短码策略。
- 完整自动化测试体系。
